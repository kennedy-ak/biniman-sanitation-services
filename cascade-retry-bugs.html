🐛 LiquidGo — Driver Cascade Bug Report

3 bugs found in the matching cascade + retry flow  |  Branch: feat/design-updates

Bug #1 — High

Retry button appears to do nothing

backend/requests_app/views.py → retry_request() line 256
What happens

When the customer clicks "Find me a driver" on an UNFULFILLED request, the view calls _dispatch_next_batch(sr) synchronously. If no driver is found right now, the cascade marks the request UNFULFILLED again and returns that status to the frontend. The frontend receives UNFULFILLED, stops polling (it's terminal), and the button reappears. From the customer's view — nothing happened.

Code path
1Customer clicks "Find me a driver" → POST /requests/57/retry/
2View resets status to PENDING, deletes old assignments
3Calls _dispatch_next_batch(sr) synchronously (blocking)
4No drivers online within 15 km → candidate_drivers() returns []
5mark_unfulfilled() runs — status back to UNFULFILLED
6Response returns UNFULFILLED → frontend stops polling → looks identical to before click
Fix

Replace the synchronous call with start_cascade.delay(sr.id) (Celery async). Return the PENDING response immediately. The frontend resumes its 3-second polling loop and shows "Finding driver…" while Celery searches in the background.

Bug #2 — Critical

Driver completes job but receives no payout after a retry

backend/requests_app/services/matching.py → mark_unfulfilled() line 219
What happens

When a request becomes UNFULFILLED, mark_unfulfilled() immediately fires task_refund_request.delay(). The payment transitions from SUCCEEDED → REFUNDED. If the customer then retries and a driver is found, the driver completes the job — but trigger_payout() checks payment.status == SUCCEEDED and finds REFUNDED instead. Payout is silently skipped. The customer has already received their refund and gets the service for free. The driver works for nothing.

Bug timeline
1Request goes UNFULFILLED → task_refund_request.delay(pk) queued immediately
2Celery runs task → payment becomes REFUNDED ← money sent back to customer
3Customer clicks retry → driver found → driver accepts, drives to location, completes job
4task_trigger_payout fires → trigger_payout() checks payment status
5Payment is REFUNDED, not SUCCEEDED → payout silently skipped 🚨
Fix

In mark_unfulfilled(), delay the refund task by 1 hour using apply_async(countdown=3600) instead of delay(). This gives the customer a grace window to retry before the refund is issued. Payment stays SUCCEEDED during the retry window, so payout works correctly if a driver is found.

Bug #3 — Medium

Refund task fires even after a successful retry

backend/payments/tasks.py → task_refund_request() line 13
What happens

task_refund_request never checks whether the request is still UNFULFILLED before calling refund_request(). If a retry succeeds (request moves to ACCEPTED or COMPLETED) while the delayed refund task is sitting in the Celery queue, the task will fire anyway and refund a payment that belongs to an active or completed job.

Code path
1Request UNFULFILLED → refund task queued with 1 h countdown (after Bug #2 fix)
2Customer retries within 1 h → driver found → request moves to ACCEPTED → COMPLETED
31 h elapses → task_refund_request wakes up in the Celery queue
4No status check → refund_request() fires → COMPLETED job's payment is refunded 🚨
Fix

Add a guard at the top of task_refund_request: re-fetch the request and return early if sr.status != 'unfulfilled'. The refund only fires if the request is genuinely still unfulfilled when the task runs — a successful retry cancels it automatically.
