What Broke & How We Fixed It

The driver matching cascade — why it failed and what we did to prevent it forever.

Background

How driver matching is supposed to work

When a customer pays for a job, this exact sequence should happen automatically:

1
Customer pays on PaystackPaystack charges the card or MoMo and confirms payment.
2
Payment confirmed in our databaseOur backend marks the payment as succeeded.
3
Cascade startsA background job (start_cascade) finds the nearest available, approved, online drivers and sends them the job offer.
4
Driver acceptsFirst driver to accept gets the job. Others are skipped.
5
If no driver responds in 60 secondsNext batch of drivers gets offered the job. If everyone is exhausted, request goes to UNFULFILLED and the customer is refunded.
The Problem

What happened with Request #44

The customer paid GHS 167.49. Payment was confirmed. But the driver never got the job offer. Here is exactly what went wrong — two separate bugs:

Bug #1 — Wrong Paystack Webhook URL

Paystack Dashboard
The webhook was pointing to bini.vendlyghana.space/... — an old domain that no longer exists. Paystack was sending payment events into the void.
Result
Paystack never successfully notified our server that the payment succeeded. Our server could not start the cascade from the webhook.

Bug #2 — Wrong Webhook Secret

.env file on server
PAYSTACK_WEBHOOK_SECRET was set to the webhook URL itself — not the actual signing secret. Every webhook Paystack sent would fail our signature check and be rejected with a 400 error.
Result
Even if the URL was right, the signature check would have failed. Double protection, both broken.

Bug #3 — No Safety Net in Code

Code (orchestrator.py)
The payment was eventually confirmed via the customer redirect (Paystack sends customers back to our site). At that point our code called start_cascade.delay(). But if that call silently failed — for any reason — there was no retry, no alarm, no recovery. The request just sat at "Finding Driver" forever.
Result
Request #44 was stuck at PENDING in the database with status "Finding Driver" on screen. It would never move forward or backward. The driver saw nothing. The customer saw nothing change.
The Fixes

What we changed — 3 things

Fix #1 — Paystack Dashboard (Config)

Before

Webhook URL: bini.vendlyghana.space/... (dead domain)

Webhook Secret: set to the URL string (wrong)

After

Webhook URL: binimansanitation.com/api/v1/payments/webhook/

Webhook Secret: blank (code falls back to the correct secret key)

Now Paystack can actually reach our server and our server trusts the message.

Fix #2 — Reliable Cascade Dispatch (Code)

In payments/services/orchestrator.py, the line that queues the cascade task (start_cascade.delay()) was moved inside a transaction.on_commit() callback.

Before

Cascade was queued immediately after saving the payment. If anything went wrong (network hiccup, Redis blip), the error was silently swallowed. No log. No retry. Gone.

After

on_commit guarantees the task is only queued after the database transaction fully commits. If it fails, a clear error is logged telling exactly which request is stuck and how to fix it.

Fix #3 — Automatic Recovery Every 2 Minutes (Code + Deploy)

A new background task called recover_stuck_cascades runs every 2 minutes via Celery Beat. It looks for any request where:

  • The payment is marked succeeded
  • The request is still pending (cascade never ran)
  • It has been more than 2 minutes since payment (grace period)

For any match it finds, it automatically re-queues the cascade. No human needed. Even if the cascade fails to fire on the first try due to any infrastructure issue, it will self-heal within 2 minutes.

The deploy script (start.sh) now also starts a celery beat process alongside the worker so this schedule actually runs on the server.

Impact

What this means at scale

🛡️
No More Silent Failures
If cascade dispatch fails, it is logged with the exact request ID. Nothing disappears quietly.
⏱️
Max 2-Minute Recovery
Even in the worst case — cascade fails on first try — the recovery job picks it up within 2 minutes automatically.
🔗
Webhooks Now Work
Paystack can now reach our server and we verify signatures correctly. The primary payment confirmation path is restored.

For thousands of users, this means: a customer pays → within seconds a driver gets the offer → within 2 minutes at worst the system self-corrects if anything hiccuped. No customer is left stuck at "Finding Driver" indefinitely.
